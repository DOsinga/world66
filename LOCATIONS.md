## Standard section structure

Every city should have the same sections in the same order. Ordering is controlled by the `SECTION_ORDER` dict in `guide/models.py` â€” no `order` field needed in the `.md` files.

| Slug | Title | Notes |
|------|-------|-------|
| `day_guides` | Curated Itineraries | Blog-sourced itineraries |
| `things_to_do` | Things to Do | Sights + museums merged, with category filters |
| `eating_out` | Eating Out | |
| `bars_and_cafes` | Bars and Cafes | Covers bars, cafes, and nightlife |
| `shopping` | Shopping | Only if there is real content |
| `day_trips` | Day Trips | |
| `beaches` | Beaches | Only where relevant (coastal cities) |
| `when_to_go` | When to Go | |
| `getting_there` | Getting There | |
| `getting_around` | Getting Around | |

**Do not use separate `sights` and `museums` sections.** Consolidate everything into `things_to_do` with category filters. If a city has existing `sights/` and `museums/` directories, migrate them (see below).

**Do not use a separate `nightlife` section.** Nightlife POIs (clubs, late-night bars) go in `bars_and_cafes` alongside bars and cafes. If a city has an existing `nightlife/` directory, move any good POIs into `bars_and_cafes/` and add a redirect. Most old World66 nightlife POI data is outdated junk â€” delete it rather than migrating it.

### What to delete

When cleaning up a city, remove these junk sections that were generated by the original World66 CMS:

- `sights.md`, `museums.md` â€” replaced by `things_to_do`
- `nightlife.md` â€” replaced by `bars_and_cafes` (delete old POIs; they're almost always outdated)
- `top_5_must_dos.md`, `budget_travel_idea.md`, `family_travel_idea.md`
- `practical_informat.md`, `7_day_itinerary.md`, `history_1.md`
- `festivals.md` (content belongs in `when_to_go`)
- `cybercafs.md`, `webcams.md`
- Any duplicates: `nightlife_and_ente.md`, `museums_1.md`, `day_trips_1.md`, etc.

**Keep `books.md`** â€” but replace the old travel-guide content with literary recommendations (see below).

### Books â€” novels, not travel guides

Every city should have a `books` section containing novels and literature that help understand the place â€” its history, its people, its particular character. **Not travel guides, not history books.** The test is: would a traveller who reads this before or during their trip understand the city differently?

Each book is a POI file in `books/` with `author:` and optionally `isbn:`:

```yaml
---
title: "My Brilliant Friend"
type: poi
author: "Elena Ferrante"
isbn: "978-1609450786"
---

The first of Ferrante's four Neapolitan novels... [2-4 sentences on what the book is and what it reveals about the city]
```

Aim for 3â€“5 books per city. Prefer novels set in the city or its region; short story collections and narrative non-fiction (Saviano's *Gomorrah*, Levi's *Christ Stopped at Eboli*) are fine when they're genuinely illuminating. Avoid: guidebooks, coffee-table books, general Italian history, books only tenuously connected to the city.

Add redirects in `redirects.json` for all deleted sections:
```json
"europe/italy/city/sights":    "europe/italy/city/things_to_do",
"europe/italy/city/museums":   "europe/italy/city/things_to_do",
"europe/italy/city/nightlife": "europe/italy/city/bars_and_cafes"
```

## The `things_to_do` section with category filters

All sights, museums, galleries, and notable neighbourhoods go in a single `things_to_do` section. The filter bar (All / Sight / Museum / Architecture / Neighbourhood) is rendered automatically â€” no code changes needed.

### Setting it up from scratch

**1. Create the section file** at `content/{path}/things_to_do.md`:

```yaml
---
title: "Things to Do"
type: section
---

Brief intro paragraph about what to see and do in the city.
```

**2. Create the POI directory** at `content/{path}/things_to_do/`

**3. Write each POI** with a `category` field and coordinates:

```yaml
---
title: "Colosseum"
type: poi
category: "Sight"
address: "Piazza del Colosseo 1, Rome"
opening_hours: "Daily from 09:00"
admission: "â‚¬16â€“â‚¬22"
latitude: 41.8902
longitude: 12.4922
tags: ["48 Hours in Rome"]
---

Description text...
```

**Recommended category values** (capitalised, consistent):

| Category | Use for |
|----------|---------|
| `Sight` | Monuments, squares, churches, ancient sites, viewpoints |
| `Museum` | Art galleries, history museums, science museums |
| `Architecture` | Buildings valued primarily for their design (not open as museums) |
| `Neighbourhood` | Districts and areas worth wandering â€” canals, market streets, etc. |

### Neighbourhood POIs (large cities)

For large cities, add `Neighbourhood` POIs for the most characterful districts â€” typically 3â€“5 per city. These work like any other POI in `things_to_do/`, but they have a special feature: **other POIs tagged with `neighbourhood: "Name"` are automatically listed on the neighbourhood page**.

This means a visitor reading the Trastevere page will see all the restaurants and bars in Trastevere listed inline â€” without Trastevere being a separate location in the hierarchy.

**How to set it up:**

1. Create the neighbourhood POI in `things_to_do/` with `category: "Neighbourhood"`:
```yaml
---
title: "Trastevere"
type: poi
category: "Neighbourhood"
latitude: 41.8893
longitude: 12.4703
---

Description of the neighbourhood...
```

2. Add `neighbourhood: "Trastevere"` to any POI (in `eating_out/`, `bars_and_cafes/`, etc.) that belongs to the neighbourhood:
```yaml
---
title: "Bar San Calisto"
type: poi
neighbourhood: "Trastevere"
...
---
```

The neighbourhood name in `neighbourhood:` must match the neighbourhood POI's `title` exactly. The matching is case-sensitive.

### Migrating from existing `sights/` and `museums/` directories

When a city already has `sights/` and `museums/` with POIs:

1. Create `things_to_do/`
2. Copy each POI file in, inserting `category: "Sight"` or `category: "Museum"` after the `type: poi` line
3. Check every POI for missing or wrong coordinates â€” fix before copying (see Coordinates section below)
4. Delete junk POIs: sports venues, commercial tours, empty entries, wrong-country content
5. Check for missing major sights â€” add them
6. Delete old `sights/` and `museums/` directories and their `.md` section files
7. Add redirects for all old URLs

```json
"europe/italy/city/sights":           "europe/italy/city/things_to_do",
"europe/italy/city/museums":          "europe/italy/city/things_to_do",
"europe/italy/city/sights/some_poi":  "europe/italy/city/things_to_do/some_poi",
"europe/italy/city/museums/some_poi": "europe/italy/city/things_to_do/some_poi"
```

## Adding POIs from Curated Itineraries

When you add a Curated Itinerary (see CLAUDE.md for the full workflow), every place it mentions should become a real POI. Use the `tags` field to link the POI to the itinerary.

### Which section does the POI go in?

| Place type | Section |
|-----------|---------|
| Sight, museum, monument, gallery, neighbourhood | `things_to_do/` |
| Restaurant, trattoria, osteria, cafĂ© serving food | `eating_out/` |
| Bar, cafĂ©, gelateria, pasticceria, aperitivo spot | `bars_and_cafes/` |

### POI template with itinerary tag

```yaml
---
title: "Trattoria Mario"
type: poi
address: "Via Rosina 2/r, 50123 Florence"
opening_hours: "Monâ€“Sat 12:00â€“15:30; closed Sun"
latitude: 43.7760
longitude: 11.2530
tags: ["3 Days in Florence"]
---

Description â€” practical, opinionated, 3â€“5 sentences. What makes it worth going to,
what to order, when to go, what to watch out for.
```

- Use the **exact tag string** from the itinerary's guide entry (e.g. `"3 Days in Florence"`, `"48 Hours in Rome"`)
- A POI can appear in multiple itineraries: `tags: ["One Day in Florence", "3 Days in Florence"]`
- If the POI already exists in the section, add the tag to its existing list rather than creating a duplicate
- Always include `category` on POIs in `things_to_do/`

## Coordinates â€” required on every POI

Every POI **must** have `latitude` and `longitude`. Without them the POI won't appear on the map and the entry is incomplete.

```yaml
latitude: 41.9009
longitude: 12.4833
```

Precision to 4 decimal places is enough (roughly 10m accuracy). Look them up via OpenStreetMap (right-click â†’ "Show address") or search "latitude longitude [place name]".

**When cleaning up existing POIs**, always check for:
- Missing coordinates â€” add them
- Wrong coordinates â€” a common problem in old World66 data where coords were copy-pasted from a different city or country entirely. Verify that the lat/lng actually falls in the right city (rough check: Rome ~41.9Â°N 12.5Â°E; Paris ~48.9Â°N 2.3Â°E; London ~51.5Â°N 0.1Â°W; New York ~40.7Â°N 74.0Â°W)

**Do not publish a POI without coordinates.** If you cannot determine them, leave the file out rather than publishing with missing or guessed coords.

## Day trips â€” linking to real locations

Day trip destinations should be real locations in the hierarchy, not POIs buried under the city. Use `linked_locations:` in `day_trips.md` to list their paths:

```yaml
---
title: "Day Trips"
type: section
linked_locations:
  - europe/italy/lazio/frascati
  - europe/italy/lazio/ostiaantica
  - europe/italy/lazio/tarquinia
---

Brief overview of day trip options from the city.
```

The template automatically renders these as a table linking to the real location pages. If a destination doesn't have a location page yet, create one in the correct place in the hierarchy before adding it here.

Delete any old POI files in `day_trips/` and add redirects pointing to the real location URLs.
